## 3. Задержки (обещания vs факт)

**Метрика — voice-to-voice: «юзер замолчал → первый звук ответа».**

**Наш стек (замерено, из хэндоффа):**
- VAD конец реплики: `redemptionMs=300` мс тишины.
- STT finalize (Ink): **18–190 мс** (против ~500 мс round-trip у Whisper-batch).
- TTS TTFB (Cartesia стрим): **~90 мс**.
- WS-ветка `/chat-voice`: первое аудио **~180 мс** после первого слова мозга.
- Итог: узкое место — мозг (LLM). Аудио-тракт (STT+TTS) очень быстрый.

**Pipecat (обещания):**
- Silero локально: «на 150–200 мс быстрее удалённых VAD».
- Smart Turn: ~65 мс инференс.
- Voice-to-voice: **~1 сек медиана** (блог Modal, на ОТКРЫТЫХ моделях) — «на уровне многих проприетарных сервисов». С Cartesia/быстрым LLM было бы сопоставимо с нами.

**Расклад:** Pipecat не быстрее по природе — он даёт те же кубики (Silero, стриминг, локальный turn). Наши цифры (STT 18–190, TTS 90, first-audio 180) **конкурентоспособны и упираются в LLM**, а не в фреймворк. Заявленная у Pipecat «1 сек» — это с открытыми моделями и сетевым round-trip; наш серверлесс-эдж + Cartesia тут не проигрывает.

## 5. Что честно стоит «украсть» (не мигрируя)
**Smart Turn v3** — единственная фича с реальным UX-выигрышем. Наш `redemptionMs=300` может рубить на паузе посреди мысли (сам риск отмечен в хэндоффе). Smart Turn решает по интонации/грамматике, «договорил ли человек».
- Модель маленькая: **Whisper-tiny + классификатор, ~8M параметров, int8**, ~10–100 мс на CPU.
- 🔑 Потенциал БЕЗ Pipecat: раз она крошечная и есть в int8 — теоретически её можно поднять **в браузере через ONNX Runtime Web** рядом с нашим Silero-VAD, и заменить фикс-тишину на ML-turn, **сохранив весь наш серверлесс-стек**. Это микро-апгрейд, а не переезд.
- Репо: `github.com/pipecat-ai/smart-turn` (open, community).

## 8. 🥷 УКРАДЕНО И ВСТРОЕНО: Smart Turn v3 в браузере (2026-07-08)
Решение Pavel: «воруй и встраивай смело, образец тестовый». Сделано.

### Что это даёт
Заменяет фикс-паузу `redemptionMs=300` на **ML-решение «договорил ли гость»** по просодии. Меньше обрубов на паузе посреди мысли.

### Как встроено (без переезда с serverless!)
- Новый модуль **`components/jarvi/jarvi-smart-turn.ts`** — Whisper log-mel + `onnxruntime-web` (уже был в проекте), модель **`public/models/smart-turn-v3.2-cpu.onnx`** (int8, 8.7МБ, кэш браузера). Рецепт 1:1 с pipecat `inference.py`: последние 8с → zero-mean-unit-var → log-mel [80,800] → ONNX → prob, порог 0.5.
- В `jarvi-engine.ts`: тумблер `config.turn` (`fixed`|`smart`), ленивая загрузка модели в фоне при старте, логика **hold/merge**: VAD дал паузу → Smart Turn; `prob≥0.5` → коммит; `prob<0.5` → держим реплику, ждём продолжения (speechStart отменяет hold) либо safety-коммит через `TURN_HOLD_MAX_MS=2500`. Склейка мид-пауз → Whisper на всей склейке (Ink-транскрипт неполон); одиночный сегмент → быстрый Ink.
- **Фоллбэк железный:** модель не загрузилась / предсказание > `TURN_PREDICT_TIMEOUT_MS=500` / любой сбой → поведение = текущая фикс-пауза. Регресс невозможен.

### ✅ Верификация (сделана БЕЗ браузера, чтобы не гадать)
Подняли Python-эталон (`transformers.WhisperFeatureExtractor(chunk_length=8)` + `onnxruntime`), прогнали их рецепт; JS-mel сверили в Node: **max abs diff 2.8e-6**, выход ONNX на JS-фичах = эталону **до 0.0**. Математика байт-в-байт. (скрипты: scratchpad `reference.py`, `mel_test.mjs`).
