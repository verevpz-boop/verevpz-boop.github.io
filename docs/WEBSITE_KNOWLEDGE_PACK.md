# ЧАСТЬ I. ДИЗАЙН- И АНИМАЦИОННАЯ ДОКТРИНА (самое ценное, портативно)

Источник — **Эмиль Ковальски** (Emil Kowalski, [emilkowal.ski](https://emilkowal.ski)): design engineer
в Linear, ex-Vercel, автор библиотек **Sonner** (тосты) и **Vaul** (iOS-style drawer). Один из эталонных
авторитетов по «дорогим» микроанимациям интерфейса. Ниже — его доктрина, которую Павел держит как закон.

## I.1. Железные правила (всегда соблюдать)

- **Никакого `linear` easing** для UI. Только spring или ease-out кривые. Линейность ощущается «мёртвой».
- **UI-анимации ≤ 150–300 мс.** Только hero/декор может быть дольше. Длинная анимация на кнопке = лаг.
- **`ease-out` по умолчанию** — быстрый старт, медленный финиш = ощущение мгновенного отклика.
- **`ease-in` почти всегда неправильно** для UI — ощущается как подвисание.
- **Анимировать ТОЛЬКО `transform` + `opacity`.** Никогда `width/height/top/left/margin/padding` —
  они дёргают layout + paint и роняют 60fps. `transform: translate/scale/rotate` — на GPU, дёшево.
- **Стартовый масштаб ≥ 0.9** — никогда `scale(0)`. Бери 0.92–0.95. Появление «из ничего» выглядит дёшево.
- **`button:active { transform: scale(0.97) }`** — глобально на всех кликабельных элементах. Тактильность.
- **`@media (prefers-reduced-motion: reduce)`** — ОБЯЗАТЕЛЬНО, не опционально. Доступность + люди,
  которых укачивает. (Готовый блок CSS — в части IV.)
- **У каждой анимации должна быть причина.** Анимация ради анимации — мусор и удешевление.

## I.2. Easing (кривые)

- **Дефолтный токен:** `cubic-bezier(0.32, 0.72, 0, 1)` — «iOS-кривая» из Vaul. Универсально красивая.
- Объект **едет по экрану** (виден всё время) → `ease-in-out`.
- Объект **появляется/исчезает** → `ease-out`.
- **Hover, смена цвета** → встроенный `ease` норм.
- Всё остальное → своя `cubic-bezier` (подбирать на [easings.co](https://easings.co)).

## I.3. Интерактивность

- **Spring для всего, что привязано к курсору** — никогда не биндить `mouseX/Y` напрямую в transform
  (дёргано). Использовать `useSpring()` (Framer Motion): stiffness ~50, damping ~20. Объект «догоняет»
  курсор с инерцией — это и есть «живое».
- **Origin-aware:** `transform-origin` из точки действия (где кликнули/навели), не из центра.
- **Прерываемость:** закрытие на полпути открытия должно плавно развернуться обратно. CSS-transitions
  и Framer Motion это умеют; CSS `@keyframes` — НЕТ (они доигрывают до конца). Для интерактивного —
  transitions/Motion, не keyframes.

## I.4. Приёмы профи

- **Blur маскирует разрывы кроссфейда** — `filter: blur(2px)` в середине перехода прячет визуальные швы.
- **Тултипы:** первый с задержкой ~500 мс, последующие мгновенно (паттерн Radix `data-instant`).
- **`clip-path`** для синхронной подсветки + смены цвета текста (паттерн вкладок Stripe).
- **CSS-анимации > JS-анимации**, когда main-thread занят (они hardware-accelerated, не зависят от RAF).

## I.5. Чего НЕ делать

- Не вешать встроенные `ease-in-out`/`ease-out` на **важные** анимации — они слабые. Своя cubic-bezier.
- Не биндить курсор в transform без spring.
- Не ставить длинные переходы на часто используемую навигацию.
- **Не тащить визуальный стиль Linear/Vercel.** Брать их анимационную дисциплину — да; их «техно-минимал»
  внешний вид — нет, если у проекта другая (например, кинематографичная/люксовая) эстетика.

## I.6. Однострочник на каждую сессию кода

> Linear easing запрещён. UI < 300 мс. Старт scale ≥ 0.9. Только transform+opacity. ease-out по умолчанию.
> Origin-aware. Reduced-motion обязателен. Spring для всего живого. Blur, если кроссфейд швом. У каждой
> анимации — причина.

## II.2. Хостинг и деплой — **GitHub Pages** (бесплатно)

- Статический экспорт (`output: "export"` в `next.config.ts`) → чистые HTML/JS/CSS, **бэкенда нет**.
- Репозиторий вида `<логин>.github.io` → сайт на корне (без `basePath`). Иначе — `<логин>.github.io/<репо>/`
  и нужен `basePath: "/<репо>"` в конфиге.
- **Авто-деплой:** push в `master`/`main` → GitHub Actions собирает → публикует. ~1–2 минуты.
- **Почему не Vercel/Cloudflare Pages:** в РФ нестабильны/режутся. GitHub Pages — стабильный бесплатный
  путь. (Для жены вне РФ-контекста Vercel может быть даже проще — реши по ситуации.)
- **Про «взлом»:** статика нечего взламывать — нет бэкенда, БД, логинов, ключей. Публикуется ровно то,
  что браузер и так качает. Публичный репо → Pages бесплатны; приватный → только на платном Pro.
- `next.config.ts` для статики: `output:"export"`, `images:{unoptimized:true}` (Pages не оптимизирует
  картинки на лету), `trailingSlash:true` (каждая страница = папка/index.html, надёжнее на статике).

## II.3. Хранилище тяжёлых медиа — **Cloudflare R2** (бесплатный тир)

- Видео/большие ассеты **НЕ** держать в git-репозитории (раздувает, тормозит) — лить на **R2**
  (S3-совместимое хранилище Cloudflare, щедрый бесплатный тир, без egress-платы).
- Публичный URL вида `https://pub-<hash>.r2.dev/<путь>`. Встраивать видео прямым `<video src=...>`.
- Заливка через `wrangler` (CLI Cloudflare). **🔴 всегда с флагом `--remote`** — иначе льёт в локальную
  эмуляцию, а не на боевой бакет.
- **CDN-Worker (`*.workers.dev`) в РФ НЕ работает** — режется ТСПУ (провайдерская фильтрация). Для CDN
  нужен **кастомный домен**, не `workers.dev`. (Вне РФ — workers.dev ок.)

## II.4. Медиа-инструменты

- **FFmpeg / FFprobe** — перекодирование и анализ видео (см. грабли про кодек ниже). Незаменимы.
- **gltf-transform** (`@gltf-transform/cli`) — сжатие 3D-моделей GLB (webp-текстуры + meshopt-геометрия).
  Ужимает модели в 4–6 раз без потери ригов. drei декодирует meshopt из коробки.
- **Spline** — визуальный редактор 3D, откуда выгружаются GLB с именованными суставами/ригами.

## II.5. «Мозг» для интерактива (чат-боты, говорящие аватары)

- **Cloudflare Worker** (бесплатный тир) как публичный прокси к LLM — ключи живут в секретах Worker'а,
  **никогда в клиентском коде**. Постоянный адрес `<имя>.<сабдомен>.workers.dev`.
- **Failover-цепочка моделей** внутри Worker'а: если одна бесплатная LLM легла — следующая. (Павел:
  Cerebras → Groq → Mistral → Gemini.) Урок: **никогда не вешать продакшн на одну бесплатку без резерва.**
- Голос: браузерный `speechSynthesis` (мгновенно, бесплатно, но робот) либо локальный нейро-TTS
  (Silero/Piper) для качества. Это уже продвинутый уровень — для типового сайта не нужно.

# ЧАСТЬ III. ВЫСТРАДАННЫЕ ГРАБЛИ (читать как «чего не делать»)

> Каждый пункт — это потерянные часы отладки. Многие универсальны для любой платформы.

## III.1. 🔴🔴 КОДЕК ВИДЕО — только **H.264**

- Chrome и Firefox **НЕ играют H.265/HEVC** в HTML5-видео → **чёрный экран**. Safari играет — поэтому
  легко не заметить на Mac и словить на проде у пользователей.
- **Перед публикацией каждого видео проверять кодек:**
  `ffprobe -v error -select_streams v:0 -show_entries stream=codec_name -of csv=p=0 <файл>`
- Если `hevc` — перекодировать в H.264:
  ```
  ffmpeg -i in.mp4 -vf "scale='min(1920,iw)':-2" -c:v libx264 -profile:v high -pix_fmt yuv420p \
    -maxrate 5M -bufsize 10M -crf 23 -preset medium -c:a aac -b:a 128k -movflags +faststart out.mp4
  ```

## III.2. 🔴 Битрейт и мобильная доставка видео

- **Корень фризов на телефоне — тяжёлый битрейт.** Видео по 15–17 Мбит/с (CRF без потолка раздувает на
  «busy» AI-картинке) грузятся еле-еле. Веб-норма: **≤ 5 Мбит/с 1080p**. Резать `-maxrate 5M`.
- **`-movflags +faststart`** обязательно — двигает moov-атом в начало файла, чтобы видео начинало
  играть до полной загрузки. Без него — долгое «висит чёрным».
- **iOS лимитирует одновременный декод видео** (`ConcurrentPlaybackNotPermitted`) — много autoplay-видео
  на странице = часть не стартует. Решение: менеджер воспроизведения (≤2 одновременно на мобиле, ≤4 на
  десктопе), `preload="none"`, выгрузка `src` за экраном через IntersectionObserver.
- **Постеры (первый кадр как JPG)** — показывать, пока видео грузится, чтобы не было чёрных дыр:
  `ffmpeg -ss 0.5 -i <файл> -frames:v 1 -q:v 4 -vf "scale='min(720,iw)':-2" poster.jpg`
- **Двухфайловая доставка** (бюджетная альтернатива HLS): лёгкое превью 480p (~0.7 Мбит/с) для автоплея +
  полный 1080p по клику. Решает ~90% проблемы за 20% усилий. Полный HLS (360/720/1080 + master.m3u8 +
  hls.js) — если канал совсем слабый/переменный.
- **Играет только видимое:** через IntersectionObserver `play()`/`pause()` — за кадром пауза экономит
  CPU и трафик.

## III.3. 🔴 CORS на медиа-хранилище

- Если видео рендерится как **WebGL-текстура** (`crossOrigin="anonymous"`) — без CORS-заголовков на
  бакете оно чёрное/висит. Обычным `<video>` на странице CORS не нужен.
- CORS на R2 ставится на бакет с allowlist origin'ов (домен сайта + localhost). **При смене домена —
  обновить список origin'ов**, иначе текстуры почернеют.

## III.4. 🔴 Грабли сборки и Windows

- `next build` на Windows может падать в финале `EBUSY: rmdir 'out'`, если висит dev-сервер. **Компиляция
  при этом прошла** (`✓ Compiled successfully`). На Linux-Actions (т.е. при push) этой ошибки нет — пуш
  собирается чисто. Локально для проверки TS достаточно увидеть `✓ Compiled`.
- **PowerShell 5.1 + кириллица в путях/именах:** `.ps1`-скрипты с кириллицей читаются как кракозябры,
  если сохранены без BOM. Сохранять **UTF-8 с BOM**.
- **`wrangler secret put` на Windows** приклеивает `\r` к значению через pipe (`"key" | wrangler...`) →
  ключ битый → 401. Лечение: `wrangler secret bulk secrets.json`.
- **r2.dev рвёт длинные закачки** → файл обрезается → `ffmpeg: moov atom not found`. После скачивания
  проверять `ffprobe ... format=duration`; пусто → перекачать.

## III.5. 🔴 Грабли 3D / WebGL (если будет R3F)

- **Не использовать `<Environment preset=.../>` из drei** — он тянет HDR с внешнего CDN (pmndrs); в РФ
  флапает и при ошибке загрузчика **рушит весь Canvas в черноту без ошибки в консоли**. Металл/отражения
  делать настройкой материала (`metalness`, `roughness`, `emissive`) + лампами сцены.
- **React 19: не создавать `<video>`-элемент в `useMemo`** — отброшенный конкурентный рендер уже начал
  качать файл (ассеты уходят в сеть дважды). Только `useEffect` + `useState`.
- **Ramp громкости зажимать `Math.max(0, Math.min(1, x))`** — выход за [0,1] бросает `IndexSizeError`
  каждый кадр и роняет WebGL (Context Lost → чёрный экран).
- **Сжимать GLB через gltf-transform** (`webp` + `meshopt`) — модели по 11 МБ убивают первый заход.
  Использовать `webp` + `meshopt`, но НЕ `optimize` (join/simplify может убить именованные суставы рига).
- **Зависший GPU-процесс Chrome** от многих перезагрузок одной вкладки → все R3F-сцены чёрные, консоль
  чистая. Это не баг кода — артефакт браузера. Проверять рендер в свежем браузере (Playwright поднимает
  свой), чинится полным перезапуском Chrome.

## III.6. 🔴 Доступность и универсальные мелочи

- `prefers-reduced-motion` — обязательный media-query (блок в части IV). Иначе укачивает и это провал
  по доступности.
- `getVoices()` (Web Speech) асинхронный — первый вызов пуст, ждать событие `voiceschanged`.
- Любой autoplay со звуком и микрофон в Chrome требуют **жеста пользователя** (user activation) —
  «открыл страницу и сразу заиграло со звуком» невозможно. Нужна стартовая кнопка.

# ЧАСТЬ IV. ЭСТЕТИКА И ГОТОВЫЕ ПАТТЕРНЫ

## IV.1. Принцип вкуса

- Эстетика Павла: **«люкс-мода Vogue, сдержанные дорогие анимации — НЕ ночной клуб».** Дорого = сдержанно,
  не «мигает и крутится». Это применимо к большинству серьёзных сайтов.
- **Палитра «тёплый нуар»** (его выбор): не чисто-чёрный `#0A0A0A`, а тёплый эспрессо `#14100B` —
  склеивается с золотом, страница «теплее» и дороже. Урок: **чистый чёрный часто выглядит дёшево;
  тёплый тёмный — благороднее.**

| Токен | Значение | Роль |
|---|---|---|
| Фон | `#14100B` | тёплый нуар (эспрессо) |
| Текст | `#F5F1E8` | тёплый белый (не чисто-белый) |
| Акцент | `#C9A961` | золото |
| Доп. | `#3D1F1F` | бордо |

- **Шрифтовая пара:** дисплейный с засечками (Cormorant Garamond) для заголовков + чистый гротеск
  (Geist Sans) для текста. Контраст «люкс-засечки + современный гротеск» = дорого и читаемо.
- **Референсы вкуса:** [romanjeanelie.com](https://www.romanjeanelie.com) (движение, тёмный люкс),
  [ui.aceternity.com](https://ui.aceternity.com) (компоненты).

## IV.2. Готовый CSS-фундамент (копировать в любой проект)

```css
:root {
  --background: #14100B;            /* тёплый нуар вместо чистого чёрного */
```

# ЧАСТЬ V. PLAYBOOK — как разобрать и улучшить сайт жены

> Это твой рабочий чек-лист. Сначала **пойми**, потом **измерь**, потом **советуй**, и только потом — правь.

## V.1. Интейк (выясни до любых советов)

1. Платформа и есть ли доступ к исходникам (или только публичный URL).
2. Цель сайта и аудитория. Что для жены значит «лучше» (быстрее / красивее / больше заявок / мобайл)?
3. На чём конкретно боль сейчас.
4. Бренд: есть ли палитра/шрифты/логотип, которых надо держаться.

## V.2. Аудит (пройди по пунктам, отметь нарушения)

**Скорость / производительность**
- [ ] Открой сайт на «медленном 3G» в DevTools и на реальном телефоне. Что грузится дольше 3 сек?
- [ ] Тяжёлые картинки (>300 КБ)? → сжать, отдать в WebP/AVIF, задать `width/height` (против layout shift).
- [ ] Видео: кодек H.264? битрейт ≤5 Мбит/с? `+faststart`? есть постер? автоплеев не слишком много?
- [ ] Шрифты грузятся с `font-display: swap`? Не висит ли текст невидимым?
- [ ] Lighthouse-прогон (Performance / Accessibility / Best Practices / SEO) — зафиксируй цифры ДО.

**Анимации (по доктрине части I)**
- [ ] Есть `linear` easing на UI? → заменить на ease-out / spring.
- [ ] Анимируются `width/height/top/left/margin`? → переписать на `transform`/`opacity`.
- [ ] UI-переходы длиннее 300 мс? → укоротить.
- [ ] Есть `prefers-reduced-motion`? Нет → добавить (блок выше). Это и доступность, и баллы Lighthouse.
- [ ] Кнопки дают тактильный `:active scale`? Курсорные эффекты на spring, не напрямую?

**Эстетика**
- [ ] Чисто-чёрный/чисто-белый? → предложить тёплые тёмные/тёплый белый, если к месту.
- [ ] Шрифтовая иерархия читаемая, есть контраст дисплейный/текст?
- [ ] Консистентны ли отступы, размеры, цвета (нет ли «зоопарка»)?

**Мобайл и доступность**
- [ ] Всё работает и красиво на 360–414px шириной? Нет горизонтального скролла?
- [ ] Тач-цели ≥ 44px? Контраст текста проходит (WCAG AA)?
- [ ] `alt` у картинок, осмысленные заголовки h1→h2→h3, фокус-стейты у интерактива.

**SEO / мета (быстрые победы)**
- [ ] `<title>` и `meta description` у каждой страницы? Open Graph (превью в мессенджерах)?
- [ ] Один `<h1>` на страницу? Человекочитаемые URL?

## V.3. Как советовать

- **Сначала измерь, потом меняй.** Зафиксируй Lighthouse/время загрузки ДО — чтобы показать жене эффект.
- **Приоритизируй по «эффект / усилие».** Сначала дешёвые большие победы (сжать видео/картинки,
  reduced-motion, постеры, мета-теги), потом дорогие переделки.
- **Уважай её бренд и платформу.** Не предлагай переезд на Next.js, если она счастлива на Tilda и сайт
  свою задачу выполняет. Доктрина анимаций/эстетики применима и там.
- **Каждое изменение — с причиной** («это уберёт фриз на телефоне», а не «так модно»).
- **Не трогай чужие аккаунты/ключи.** Всё новое — на её собственных аккаунтах.

## V.4. Если жена захочет «премиум как у мужа»

Тогда применим стек части II: GitHub Pages (бесплатно) + R2 для медиа + Next/R3F при желании 3D. Но это
переезд — обсуждать отдельно, с её согласия, поэтапно. Грабли частей III обязательны к прочтению ПЕРЕД
стройкой, а не после.

# ПРИЛОЖЕНИЕ. Шпаргалка команд

```bash
# Проверить кодек видео
ffprobe -v error -select_streams v:0 -show_entries stream=codec_name -of csv=p=0 video.mp4

# Перекодировать в веб-H.264 ≤5 Мбит/с с faststart
ffmpeg -i in.mp4 -vf "scale='min(1920,iw)':-2" -c:v libx264 -profile:v high -pix_fmt yuv420p \
  -maxrate 5M -bufsize 10M -crf 23 -preset medium -c:a aac -b:a 128k -movflags +faststart out.mp4

# Постер (первый кадр) из видео
ffmpeg -ss 0.5 -i video.mp4 -frames:v 1 -q:v 4 -vf "scale='min(720,iw)':-2" poster.jpg

# Сжать 3D-модель GLB (webp-текстуры + meshopt-геометрия)
npx @gltf-transform/cli webp in.glb tmp.glb
npx @gltf-transform/cli meshopt tmp.glb out.glb

# Залить медиа на Cloudflare R2 (ВСЕГДА --remote!)
npx wrangler r2 object put "bucket/key.mp4" --file=local.mp4 --content-type="video/mp4" --remote
```
